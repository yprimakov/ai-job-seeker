# Project Description Template

A unified structure for documenting development projects on yuryprimakov.com. Use this outline when writing or revising project long descriptions in the admin portal.

---

## How to Use This Template

- **Required sections** are marked with `*`. Include these in every project.
- **Optional sections** are context-dependent. Include them when relevant.
- Write in the site's brand voice: conversational, direct, no filler. Let the work speak.
- Headings should use `<h2>` tags in the rich text editor. Subheadings use `<h3>`.
- Keep paragraphs short. Use bullet lists for scannable content.

---

## Section Outline

### 1. Overview *

What the project is, who it serves, and why it exists. This is the hook. Lead with the what, then the why.

**Include:**

- What the project does in 2-3 sentences
- Who the end users or stakeholders are
- The problem it solves or opportunity it addresses
- Context (organization, initiative, or personal motivation)

**Avoid:**

- Repeating the short description verbatim
- Vague statements like "a cutting-edge solution"

---

### 2. Business Impact / Key Outcomes *

What changed because this project exists. Concrete results over abstract claims.

**For professional projects, cover:**

- Measurable improvements (time saved, users served, processes eliminated)
- Key capabilities delivered (as a bullet list)
- How it changed workflows, operations, or user experience
- Any cost savings, efficiency gains, or scale achieved

**For personal projects, cover:**

- What you learned or proved
- What the project enables for its users
- Key capabilities (as a bullet list)

---

### 3. My Role & Contributions

What you specifically did on the project. This is what differentiates a portfolio piece from a product page.

**Include:**

- Your title/role on the project (reinforces the metadata)
- Specific responsibilities and ownership areas
- Key decisions you made and their rationale
- Collaboration points (who you worked with, how teams were structured)

**Skip this section if** you were the sole developer and the role metadata already makes that clear.

---

### 4. Architecture / Technical Approach *

How the system is built. Keep it accessible to non-engineers but detailed enough for technical readers.

**Include:**

- High-level architecture (frontend, backend, integrations, infrastructure)
- Key technical decisions and why they were made
- Data flow or system interaction patterns (if noteworthy)
- Scale considerations (concurrent users, data volume, performance targets)

**Keep it concise.** A paragraph or two is usually enough. Save deep dives for blog posts.

---

### 5. Tech Stack

Explicit list of technologies used. Group by layer.

**Format:**

- **Frontend:** [frameworks, languages, styling]
- **Backend:** [runtime, framework, APIs]
- **Database:** [engine, ORM]
- **Infrastructure:** [hosting, CI/CD, cloud services]
- **Other:** [ML frameworks, third-party APIs, design tools]

**Skip this section if** the tech stack is already covered sufficiently in the Architecture section.

---

### 6. Design (optional)

Visual or UX design work worth highlighting. Include this when design was a significant part of the project.

**Include:**

- Design approach and methodology
- Key UX decisions
- Embedded mockups, wireframes, or design documents (PDF embeds, images)
- Tools used (Figma, etc.)

---

### 7. Challenges & Solutions (optional)

Technical or organizational obstacles and how you solved them. This shows problem-solving ability.

**Format as pairs:**

- **[Challenge]:** Brief description of the problem
- **[Solution]:** What you did about it

3-5 pairs is the sweet spot. Only include genuinely interesting challenges, not routine work.

---

### 8. Accolades & Recognition (optional)

Awards, press mentions, or notable adoption milestones.

**Include:**

- Award name, year, and granting organization
- What specifically was recognized
- Supporting images (award photos, press clippings)

---

### 9. Media & Documents (optional)

Embedded screenshots, videos, PDFs, or demo links that add context.

**Placement guidelines:**

- Hero images or project logos can go above the Overview
- Screenshots and diagrams go inline near the section they support
- PDF embeds (design docs, reports) go in their own section or after Design
- Video demos should be near the top for maximum engagement

---

### 10. Lessons Learned / Retrospective (optional)

Brief reflection on what you would do differently or what the project taught you. Best suited for personal projects or older professional work.

---

### 11. Future Enhancements (optional)

Planned or potential improvements. Include only if the project is actively maintained.

---

## Section Order Reference


| #   | Section                           | Required    | Notes                           |
| --- | --------------------------------- | ----------- | ------------------------------- |
| 1   | Overview                          | Yes         | Always first                    |
| 2   | Business Impact / Key Outcomes    | Yes         | Lead with results               |
| 3   | My Role & Contributions           | Recommended | Skip if solo + obvious          |
| 4   | Architecture / Technical Approach | Yes         | Keep accessible                 |
| 5   | Tech Stack                        | Recommended | Skip if covered in Architecture |
| 6   | Design                            | Optional    | When design was significant     |
| 7   | Challenges & Solutions            | Optional    | 3-5 pairs max                   |
| 8   | Accolades & Recognition           | Optional    | When applicable                 |
| 9   | Media & Documents                 | Optional    | Inline or standalone            |
| 10  | Lessons Learned                   | Optional    | Best for personal projects      |
| 11  | Future Enhancements               | Optional    | Active projects only            |
